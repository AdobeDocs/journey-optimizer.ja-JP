# ジャーニーとキャンペーン：適切なアプローチを選択 {#journeys-vs-campaigns}

Adobe Journey Optimizerには、顧客にリーチし顧客を引き付ける 3 つの強力なアプローチがあります。 効果的なマーケティングエクスペリエンスを構築するには、それぞれを使用するタイミングを理解することが重要です。

このガイドは、具体的なマーケティングのニーズに基づいて、**ジャーニー**、&lbrace; キャンペーン **&#x200B;**&#x200B;（アクションおよび API トリガー） **、オーケストレーションされたキャンペーン** から選択するのに役立ちます。

## クイック比較の概要 {#quick-overview}

| アプローチ | に最適 | 実行スタイル |
|----------|----------|-----------------|
| **ジャーニー** | 条件付きロジックを使用したマルチステップのリアルタイム顧客体験 | 1:1 オーケストレーション – 各プロファイルを自分のペースで |
| **キャンペーン（アクションおよび API）** | シンプル、スケジュール済みまたはトリガーされたメッセージ配信 | バッチまたは API トリガー – すべてのプロファイルが同時に |
| **調整されたキャンペーン** | マルチエンティティのセグメント化を使用した複雑なバッチワークフロー | バッチキャンバス – 一緒に処理されるすべてのプロファイル |

## 詳細な比較 {#detailed-comparison}

次の包括的な表を使用して、主な違いを理解します。

| 機能 | ジャーニー | キャンペーン（アクションおよび API トリガー） | 調整されたキャンペーン |
|---------|----------|-----------------------------------|----------------------|
| **プライマリの目的** | リアルタイム顧客コンテキストを使用した複数手順 1:1 オーケストレーション | オーディエンスに対する 1 回限りのまたは定期的なメッセージ配信 | 複雑なセグメント化ワークフローを使用した複数ステップのバッチキャンペーン |
| **キャンバスタイプ** | 1:1 キャンバス – 各プロファイルは独自のペースで移動します。 | キャンバスなし – シングルアクションの実行 | バッチキャンバス – 一緒に処理されるすべてのプロファイル |
| **実行フロー** | 順次アクション、プロファイルはジャーニー全体を通じて状態を維持 | オーディエンス全体への同時実行 | アクティビティとトランジションを含む複数ステップのバッチワークフロー |
| **立入メカニズム** | イベント，オーディエンス，資格，ビジネスイベント | 手動によるアクティベーション、スケジュール設定、API のトリガー | バッチワークフローのスケジュールされた実行 |
| **データモデル** | リアルタイムプロファイルとイベントデータ | プロファイルデータ + API ペイロード | マルチエンティティのリレーショナルデータ（プロファイル、製品、ストア、予約） |
| **セグメント化** | 事前定義済みオーディエンス + リアルタイム条件 | Experience Platformの事前定義済みオーディエンス | 正確なカウントを使用してキャンバス内で作成されたオンデマンドオーディエンス |
| **プロファイルの処理** | 個人、リアルタイム（イベント発生時） | バッチ、すべてを一度に | バッチ、すべてを伴うマルチエンティティのサポート |
| **パーソナライゼーション** | リアルタイムコンテキストデータ + プロファイル属性 | プロファイル属性+ API ペイロードデータ | 精度ターゲティングのためのマルチエンティティデータ |
| **複雑性** | ブランチ、待機時間、条件を含む複数のステップ | 単一のアクションまたはシンプルなワークフロー | セグメント化、エンリッチメント、分割による複数ステップのバッチワークフロー |
| **最適な対象** | カスタマーライフサイクルジャーニー、オンボーディング、買い物かごの放棄 | プロモーションキャンペーン、ニュースレター、お知らせ、トランザクションメッセージ | 複雑な季節限定キャンペーン、複数手順のプロモーション、製品ローンチ |
| **タイミング** | 継続的、公開後は常にアクティブ | スケジュールされた開始日/終了日または API トリガー | スケジュールに従ったバッチ実行 |
| **国の管理** | リアルタイムでアクションを実行できるよう顧客の状態を維持 | ステートレス実行 | ワークテーブルでのバッチ処理 |
| **用途** | リアルタイムの決定ロジックを使用した複数のタッチポイントが必要 | 特定の時間にオーディエンスに送信されるシンプルなメッセージ | 複雑なセグメント化、マルチエンティティデータ、または正確な事前送信数が必要 |
| **独自の機能** | リアルタイムの反応、待機アクティビティ、プロファイルベースのペーシング | シンプルなスケジュール設定、API トリガー、レート制御 | リレーショナルデータセット、マルチエンティティのセグメント化、正確なカウント、マルチレベルの送信 |

## 決定ガイド {#decision-guide}

適切なアプローチを選択するには、次のデシジョンツリーに従います。

### 手順 1：実行要件を教えてください。

**顧客の行動に対する個人のリアルタイムの応答**
→ **ジャーニーを使用**
- プロファイルは自分のペースで移動する必要があります
- 動作に基づく条件付きロジック
- リアルタイムコンテキストが重要です

**オーディエンスに対するシンプルなメッセージ配信**
→**アクションまたは API トリガーキャンペーンの使用**
- すべてのプロファイルが同時にメッセージを受信
- API を介してスケジュールまたはトリガー
- 複雑な複数手順のロジックは不要

**複雑なバッチワークフローと高度なセグメント化**
→**オーケストレートキャンペーンの使用**
- マルチエンティティデータ（製品、店舗、予約）が必要
- 正確な事前送信数を要求
- 分割とエンリッチメントによる複数ステップのバッチ処理

### 手順 2：選択を検証

| お客様のニーズ | 推奨アプローチ | 理由 |
|-----------|---------------------|-----|
| 複数の手順から成るオンボーディングで新規のお客様を歓迎します | ジャーニー | リアルタイムエントリ、複数のタッチポイント、条件付きパス |
| 購読者への月次ニュースレターの送信 | アクションキャンペーン | オーディエンスに対するシンプルなスケジュール済みメッセージ |
| リマインダーシーケンスを使用した買い物かご放棄 | ジャーニー | リアルタイムトリガー、待ち時間、条件付きフォローアップ |
| すべてのお客様へのプロモーションのお知らせ | アクションキャンペーン | 1 回限りのメッセージ、即時配信 |
| 行動に基づいて非アクティブユーザーを再エンゲージ | ジャーニー | オーディエンスの選定によってトリガーされ、パーソナライズされたパス |
| ビジネスイベントによってトリガーされるフラッシュ販売 | ジャーニー（ビジネスイベント） | 複数のお客様に影響するリアルタイムのトリガー |
| 商品カタログの統合による季節的なプロモーション | 調整されたキャンペーン | 複数エンティティのデータ、複雑なセグメント化、正確なカウント |
| API トリガートランザクションメッセージ | API トリガーキャンペーン | 外部システムのトリガー、即時配信 |
| 予約ごとのマルチレベル送信 | 調整されたキャンペーン | 複数エンティティの関係、予約ごとに 1 つのメッセージ |

## 主な特徴 {#key-distinctions}

### ジャーニー: 1:1 リアルタイムオーケストレーション

**ユニークな理由：**
- 各プロファイルは、個々の状態とコンテキストを維持します
- プロファイルは自分のペースで入力および進行します
- 行動とイベントに基づくリアルタイムの意思決定
- 待機アクティビティでパーソナライズされたタイミングを作成
- 条件付きブランチでは、プロファイルごとに一意のパスが作成されます

**フローの例：**

```
Customer A: Abandoned cart → Wait 2 hours → No purchase? → Send reminder → Purchased? → End
Customer B: Abandoned cart → Wait 2 hours → Already purchased → End immediately
```

各顧客は、アクションに基づいて独自のジャーニータイムラインを体験します。

[ジャーニーの詳細情報](../building-journeys/journey.md)

### キャンペーン：シンプルなバッチ配信またはトリガー配信

**ユニークな理由：**
- すべてのプロファイルが同一および同時に処理される
- ステートレス実行 – コンテキストが保持されない
- シンプルなスケジュール設定または API トリガー
- ブロードキャスト通信に最適

**フローの例：**

```
Monday 9 AM → Send newsletter to 100,000 subscribers → All receive simultaneously
```

全員が同時に同じメッセージを受け取ります。

**タイプ：**
- **アクションキャンペーン**：スケジュールされた配信（1 回限りまたは繰り返し）
- **API トリガーキャンペーン**：外部システムからの API 呼び出しを介してトリガーされます

[Campaign の詳細情報](../campaigns/get-started-with-campaigns.md)

### 調整されたキャンペーン：バッチキャンバスワークフロー

**ユニークな理由：**
- アクティビティとトランジションを含むバッチキャンバス（ジャーニーキャンバスと似ていますが、バッチ指向です）
- マルチエンティティリレーショナルデータのサポート（プロファイル +製品+店舗+予約）
- キャンバス内でのオンデマンドのオーディエンス構築
- 送信前の正確なカウント（送信前の表示）
- マルチレベル送信（エンティティごとに 1 つのメッセージ（予約ごとに例：）
- バッチで処理されたすべてのプロファイル

**フローの例：**

```
Query customers → Filter by purchase history → Split by region → 
Enrich with product data → Build segments → Send personalized offers → All in one batch execution
```

ワークフローの複雑さをバッチキャンペーンの実行と組み合わせます。

[オーケストレートキャンペーンの詳細を学ぶ](../orchestrated/gs-orchestrated-campaigns.md)

## 使用例 {#use-cases}

### ジャーニーのユースケース

- **買い物かご放棄の回復**：買い物かごへの追加イベントによってトリガーされ、チェックアウトを待ち、購入がない場合はリマインダーを送信します
- **顧客のオンボーディング**：プロファイルデータに基づいてパーソナライズされたコンテンツを含む、複数の手順から成るウェルカムシリーズ
- **ロイヤルティ層のアップグレード**：顧客が新しい層に到達したときにトリガーされ、お祝いの言葉と特典を送信します
- **誕生日キャンペーン**：誕生日に基づくエントリ、パーソナライズされたオファー
- **再エンゲージメント**：オーディエンスの選定（非アクティブ）、プログレッシブアウトリーチによってトリガーされます

### Campaign の使用例（アクションおよび API トリガー）

**アクションキャンペーン：**
- **月次ニュースレター**：購読者セグメントへのスケジュールされたバッチ配信
- **プロモーションのお知らせ**：オーディエンスをターゲットにするための、時間に依存するオファー
- **製品ローンチ**：すべてのお客様に対する調整された発表
- **季節ごとの挨拶**：特定の日にホリデーメッセージ

**API トリガーキャンペーン：**
- **注文確認**：購入後に e コマースシステムによってトリガーされる
- **出荷通知**：ロジスティクスシステムによるトリガー
- **アカウントアラート**：不正検知システムによりトリガーされます
- **パスワードのリセット**：アプリケーションのユーザーアクションによってトリガーされます

### 調整された Campaign の使用例

- **カタログ統合による季節的なプロモーション**：製品カタログのクエリ、適格な顧客の特定、好みによるセグメント化、パーソナライズされた製品レコメンデーションの送信
- **店舗固有のキャンペーン**：店舗在庫データを使用して、特定の店舗場所の近くにある顧客をターゲットにします
- **複数予約コミュニケーション**：予約ごとに 1 つのメッセージを送信します（ホテル予約、フライト予約）
- **複雑なセグメントオーケストレーション**：複数のデータソースからのエンリッチメントを使用して、オーディエンスを段階的に構築する
- **送信前検証**：主要なキャンペーンを開始する前に、受信者の正確な数を取得します

## よくある質問 {#common-questions}

+++ マーケティング戦略でジャーニーとキャンペーンを組み合わせることはできますか？

絶対！ ほとんどの組織では、次に示す 3 つの方法をすべて、異なるシナリオに対して使用しています。
- 行動リアルタイムエンゲージメントのジャーニー
- スケジュールされたブロードキャスト通信のアクションキャンペーン
- トランザクションメッセージ用の API トリガーキャンペーン
- 複雑でデータ量の多いバッチキャンペーン用にキャンペーンを調整

+++

+++ キャンペーンをジャーニーに変換したり、ジャーニーをジャーニーに変換したりできますか？

いいえ。適切な形式でエクスペリエンスを再構築する必要があります。 ただし、コンテンツ、オーディエンス、ロジック概念は再利用できます。

+++

+++ 作成が容易なアプローチはどれですか？

アクションキャンペーンは通常、最も単純なキャンペーン（単一のメッセージからオーディエンスへ）で、次に API トリガーキャンペーン、ジャーニー（複数手順のロジックでより複雑になる）、オーケストレートキャンペーン（キャンバスワークフローと複数エンティティの機能により最も複雑になる）です。

+++

+++ 大規模なオーディエンスに適したスケールはどれですか？

3 つすべてが適切に拡張できますが、
- **オーディエンスを読み取りジャーニー** および **アクションキャンペーン** は、大きなバッチオーディエンスに最適化されています
- **キャンペーンの調整** 大規模なデータセットを使用した複雑なセグメント化で優れている
- **単一ジャーニー** プロファイルを個別に処理するため、規模はイベントの量に依存します

+++

+++ ジャーニーとキャンペーン全体で同じオーディエンスを使用できますか？

はい、Adobe Experience Platformで作成したオーディエンスは、3 つのすべてのアプローチで使用できます。

+++
