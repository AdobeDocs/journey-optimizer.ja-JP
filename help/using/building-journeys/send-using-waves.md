# ジャーニーでウェーブを使用して送信 {#send-using-waves-journeys}

ジャーニーからのアウトバウンドメッセージを、一度にではなくバッチ（ウェーブ）で送信できます。 ウェーブ送信は、負荷のバランスを取り、ダウンストリームの圧倒的なシステム（コールセンターやランディングページなど）を回避し、配信品質と送信者のレピュテーション（特に大量の読み取りオーディエンスジャーニー）をサポートするのに役立ちます。

<!--
>[!CAUTION]
>
>Wave sending is available for read audience journeys only and applies to **outbound** actions only (Email, SMS, Push, Direct mail).
-->

オーディエンスの入力方法とアクションのスケジュール方法を定義する場合は、ジャーニーレベルで設定します。 波の数、波のサイズ（オーディエンスのパーセンテージまたは絶対数）、および各波の実行時間を定義します。

## 制限とガードレール {#limitations-guardrails}

* ウェーブ送信は、**[!DNL As soon as possible]**&#x200B;および&#x200B;**[!UICONTROL Once]** スケジューラータイプの読み取りオーディエンスジャーニーでのみ使用できます。 詳しくは、[&#x200B; ジャーニースケジュール &#x200B;](read-audience.md#schedule)を参照してください。
* ウェーブ送信は、繰り返し、イベントトリガー、ビジネスイベント、テストモードまたはドライランのジャーニーでは使用できません。
* 少なくとも&#x200B;**2個のウェーブ**&#x200B;を定義する必要があり、最大&#x200B;**10個のウェーブ**&#x200B;を追加できます。
* 2つのウェーブの開始の最小間隔は&#x200B;**30分**&#x200B;です。
* ウェーブ開始は、ジャーニー開始の前または過去にすることはできません。
* オーディエンスを波に分割すると、最大1時間かかる場合があります。 プロファイルはその時点までジャーニーにエントリしない場合があります。
* 1つのジャーニーバージョン内では、2つの波が同時に実行されることはありません。 次の波は、前の波が終わった後にのみ始まります。 例えば、波が1時間間隔でスケジュールされているが、第1波が2時間実行されている場合、第2波は、スケジュールされた時間ではなく、第1波が終了したときに開始されます。
* ウェーブ開始は、プラットフォームがクォータ制限を適用する場合や、システム容量が負荷の大きい場合に遅れる可能性があります。

## ジャーニーでのウェーブ送信の設定 {#configure-wave-sending}

1. [&#x200B; オーディエンスの読み取り](read-audience.md) アクティビティでジャーニーを開始します。

1. 「**[!UICONTROL オーディエンスを読み取り]**」アクティビティをダブルクリックしてプロパティを開き、「**[!UICONTROL ウェーブでジャーニーアクションを配信]**」オプションを選択します。

   ![](assets/journey-wave-option.png){width="100%"}

1. **ウェーブ数** （例：4）を設定します。

   ![](assets/journey-wave-number.png){width="80%"}

   >[!NOTE]
   >
   >少なくとも2つの波を定義し、最大10の波を追加できます。

1. 波のサイズとタイミングを定義する方法を以下に説明します。

### 等しい波 {#equal-waves}

デフォルトでは、オーディエンスは同じサイズのウェーブに分割されます。 各波の開始間隔（2時間など）を一定の間隔で設定します。

![](assets/journey-equal-waves.png){width="70%"}

>[!NOTE]
>
>2つのウェーブの開始の最小間隔は&#x200B;**30分**&#x200B;です。

次に、後続の波を自動的にスケジュールします（例えば、午前9:00、午前11:00、午後1:00、午後3:00、第3）。

### カスタム配布 {#custom-distribution}

「**[!UICONTROL カスタム分布]**」オプションを選択して、各ウェーブのサイズをオーディエンス全体に対する割合で定義します（例：15%、20%、25%、40%）。

![](assets/journey-wave-percentage.png){width="70%"}

**[!UICONTROL Numbers]**&#x200B;を選択して、各波のサイズをプロファイルの絶対数として定義します（例：10,000; 50,000）。

![](assets/journey-wave-numbers.png){width="70%"}

>[!NOTE]
>* パーセンテージを使用する場合、すべての波は合計100%である必要があります。 そうでない場合は警告が表示されます。
>* 数値を使用する場合、システムはカバレッジを検証しません。ウェーブサイズが意図したオーディエンスをカバーしていることを確認してください。 [詳細情報](#faq)

### カスタムスケジュール {#custom-schedule}

**[!UICONTROL 各波のスケジュール]**&#x200B;を選択して、各波の特定の開始日時を定義します。 波の間隔を均等にする必要はありません（例えば、午前9:00、午前11:00、午後5:00、午後8:30）。

![](assets/journey-wave-custom-schedule.png){width="70%"}

>[!NOTE]
>
>2つのウェーブの開始の最小間隔は&#x200B;**30分**&#x200B;です。

## ユースケース {#use-cases}

Wave送信は、送信するメッセージの数とタイミングを制御するのに役立ちます。これにより、配信品質を向上させ、送信者のレピュテーションを保護し、送信を運用能力に合わせることができます。 以下のシナリオでウェーブを使用することを検討してください。

* **コールセンターまたは応答管理：** 1日または1時間あたりのメッセージ数を制限して、ダウンストリーム チーム（カスタマーケアなど）が応答を処理できるようにします。 例えば、コールセンターのキャパシティに合わせて、1日あたり20通のメッセージを送信します。

  ![](assets/journey-waves-ex-call-center.png){width="55%"}

* **配信数と配信品質が多い：** 1回のショットで非常に大きなジャーニーの送信を避けます。 配信を長期的に分散することで、送信者のレピュテーションを維持し、スパムとしてフラグが立てられるリスクを低減します。

  ![](assets/journey-waves-ex-high-volume.png){width="55%"}

* **ランプアップ：**&#x200B;新しいプラットフォームまたはIPを使用する場合、徐々にボリュームを増やして（たとえば、最初の波で10%、次に15%、20%など）、徐々にレピュテーションを構築します。

  ![](assets/journey-waves-ex-ramp-up.png){width="55%"}

## よくある質問 {#faq}

+++ 波のサイズの合計がオーディエンスの合計と等しくない場合はどうなりますか？

* 波のサイズの合計がオーディエンスの&#x200B;**を超える**&#x200B;場合（例えば、100,000人のオーディエンスの最初の波で100,000人をスケジュールする）、最初の波は完全なオーディエンスに送信され、残りの波には送信する残りの1人が残りません。実行されません。
* 合計&#x200B;**がオーディエンスより小さい**&#x200B;場合（例えば、100,000人のオーディエンスに対して合計40,000個のプロファイルを含む4つのウェーブを定義する）、これらのウェーブに含まれるプロファイルのみがメッセージを受信します。 他のオーディエンスは通信を受け取らず、後の波で再試行されません。

+++

+++ 個々のウェーブに異なるセグメントや基準を割り当てることはできますか？

波のサイズとタイミングのみを定義できます。 同じオーディエンスがジャーニーを流れます。個々のウェーブに異なるセグメントや基準を割り当てることはできません。

+++
