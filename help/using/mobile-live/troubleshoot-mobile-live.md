# ライブアクティビティのトラブルシューティング {#troubleshoot-mobile-live}

Adobe Journey Optimizerのライブアクティビティを使用すると、iOSのロック画面や動的な島々を、リアルタイムで動的に更新できます。 API トリガーキャンペーンでのみトリガーおよび管理できます。

**ユースケースのタイプ：**

* **単一**：個別にターゲットされるトランザクション（API トリガーのトランザクションキャンペーン）
* **ブロードキャスト**：オーディエンスをターゲットにした大量配信（API トリガーマーケティングキャンペーン）

ライブアクティビティでよくある課題は、ライブアクティビティをトリガーまたは更新するための API 呼び出しで **成功した応答（200 OK）が返されたが** ユーザーのデバイスにライブアクティビティが表示されない、または更新されない場合です。 この API 確認と実際のデバイス動作の間の切断は、配信パイプラインの複数のポイントで発生する可能性があります。 このガイドでは、配信が失敗している場所を特定するための体系的なトラブルシューティングアプローチを提供し、API リクエストの検証からデバイスレンダリングまで、各ステージを調べます。

## 前提条件

トラブルシューティングの前に、次のことを確認します。

* &#x200B;
  +++ Assurance セッションの設定

  **Assurance セッション** を設定して、SDK イベントをキャプチャし、配信パイプラインを調べます。 Assuranceは、以下を可視化します。

   * Edge Networkのリクエストと応答
   * プロファイルの選定イベント
   * プッシュトークンの登録
   * ライブアクティビティのライフサイクルイベント

  Assuranceの設定方法については、[Adobe Experience Platform Assurance ドキュメント &#x200B;](https://experienceleague.adobe.com/ja/docs/platform-learn/implement-mobile-sdk/app-implementation/assurance) を参照してください。

  **注意**: iOSのライブアクティビティでは、アプリがiOSの物理デバイス（iOS 16.1 以降）または Xcode シミュレーター（iOS 16.1 以降）で動作していることを確認してください。

  +++

* &#x200B;
  +++ API トリガーキャンペーンの詳細の収集

  Journey Optimizerの API トリガーキャンペーンに移動して、以下を取得します。

   * キャンペーン名
   * URL またはキャンペーンプロパティで見つかったキャンペーン ID
   * キャンペーンのバージョン （該当する場合）
   * サーフェス設定、ライブアクティビティに使用されるiOS アプリサーフェス

  +++

* &#x200B;
  +++ API リクエスト情報の収集

  API 呼び出しを行ってライブアクティビティをトリガーにする場合は、次を保存します。

   * プロファイル識別子とライブアクティビティデータを含む、API リクエストペイロード
   * ステータスコード、メッセージ ID、リクエスト ID などの API 応答
   * API が呼び出されたときのタイムスタンプ
   * 使用されているエンドポイント（例：`/campaign/{CAMPAIGN_ID}/execute`）

  +++

* &#x200B;
  +++ テストプロファイルの識別

  API リクエストから、次を取得します。

   * プロファイル名前空間（ECID、メール、顧客 ID など）
   * API 呼び出しで使用されるプロファイル ID

  Adobe Experience Platformでこのプロファイルを検索できることを確認します。 方法について詳しくは [Experience Platform ドキュメントでのプロファイルの検索 &#x200B;](https://experienceleague.adobe.com/en/docs/experience-platform/profile/ui/user-guide.html) を参照してください。

  +++

* &#x200B;
  +++ デバイスとアプリの情報

  テストデバイスから次の情報を収集します。

   * デバイスモデル（例：iPhone 14 Pro）
   * iOS版
   * アプリバンドル識別子
   * APN プッシュトークン
   * テスト時のネットワーク接続ステータス

  +++

## 一般的なシナリオ

### プロファイルまたはプッシュトークンの問題 {#profile-issue}

[!BADGE &#x200B; 単一のユースケースとブロードキャストのユースケースの両方に適用 &#x200B;]{type=Positive}

この API は HTTP 200 を返しますが、ライブアクティビティは表示されません。 一般的な原因：

* プロファイルがAdobe Experience Platformに存在しません。
* ライブアクティビティのプッシュトークンがプロファイルに同期されていません。
* ライブアクティビティのプッシュの詳細が同期されますが、設定が正しくありません（`appId` や `attributeType` が間違っているなど）。

**ブロードキャストのユースケースに関する注意**：オーディエンス内の一部のプロファイルがトークンを欠落している場合、それらのプロファイルのみがライブアクティビティを受信できません。 オーディエンスから複数のプロファイルをサンプル化し、トークンの問題を診断します。 これは、リモートの開始イベントにのみ適用され、更新イベントや終了イベントには適用されません。

#### 事前確認

* iOS アプリケーションの要件：
   * iOS 16.1 以降
   * `NSSupportsLiveActivities` で `YES` が `Info.plist` に設定されました
   * `ActivityAttributes` が正しく実装されました。
* Mobile SDKの統合：
   * Adobe Experience Platform Mobile SDK（messaging SDK 5.11.0 以降）
   * `Messaging.registerLiveActivities` を実装し、ライブアクティビティのプッシュトークンで呼び出します。

#### デバッグ手順

1. &#x200B;
   +++ Adobe Experience Platformにプロファイルが存在することを確認

   1. Journey Optimizerで、**顧客**/`>` プロファイル **&#x200B;**&#x200B;移動します。
   1. API リクエストの名前空間と ID 値を使用した検索。
   1. プロファイルが見つからない場合、プロファイルが存在しないか、取り込みが完了していません。 プロファイルを作成するか、取り込みを待機してからライブアクティビティをトリガーします。
   1. プロファイルが見つかった場合は、以下の手順 2 に進んで、プッシュトークンが同期されているかどうかを確認します。

      +++

1. &#x200B;
   +++ ライブアクティビティのプッシュトークンが同期されているかどうかを確認

   Assuranceを使用してトークン登録を検証できます。

   1. Assuranceの **イベント** リストで、イベント `eventType = "liveActivity.pushToStart"` をフィルタリングまたは検索します。
   1. **イベント** を選択し、ペイロードを調べます。
   1. token、appId、attributeType の値が存在することを確認します。
   1. イベントが正常に送信されたかどうかを確認します。

   また、Adobe Experience Platform プロファイルをチェックインすることもできます。

   1. Adobe Experience Platformの **プロファイル** から「**イベント**」タブにアクセスします。
   1. `liveActivity.pushToStart` イベントを検索します。
   1. イベントのタイムスタンプとペイロードを確認します。

   イベントが見つからない場合、モバイルアプリは `Messaging.registerLiveActivity` を正しく呼び出していません。 SDK統合を修正する必要があります。

   +++

1. &#x200B;
   +++ プロファイルのトークン詳細の検証

   1. **プロファイル** から、「**属性** タブにアクセスします。
   1. `liveActivityPushNotificationDetails` を見つけます。
   1. トークン設定を確認します。

      ```json
      {
        "liveActivityPushNotificationDetails": [
          {
            "appId": "com.example.myapp",
            "token": "abc123def456...",
            "platform": "apns",
            "denylisted": false,
            "attributeType": "OrderTrackingAttributes",
            "identity": {}
          }
        ]
      }
      ```

   **各フィールドを検証する：**

   | フィールド | 要件 | 共通の問題 |
   |-|-|-|
   | `appId` | iOS バンドル ID と完全に一致する必要があります | 開発/実稼動バンドル ID が一致しません |
   | `attributeType` | Swift `ActivityAttributes` 構造体名と完全に一致する必要があります（大文字と小文字を区別） | 入力が正しくないか、構造体名が正しくありません |
   | `platform` | `"apns"` または `"apnsSandbox"` である必要があります | プラットフォーム値が間違っています |
   | `denylisted` | `false` である必要があります | トークンが無効としてマークされているか、ユーザーがオプトアウトしました |
   | `token` | 有効な APN プッシュトークン | トークンが期限切れであるか、アプリが再インストールされました |

   いずれかのフィールドが正しくない場合：モバイルアプリを更新し、`Messaging.registerLiveActivities` を使用して再登録し、5～10 分待ってから、再確認します。

   `liveActivityPushNotificationDetails` がない場合：トークンがまだ同期されていません。 Assuranceで `liveActivity.pushToStart` イベントを確認してから 5～10 分待ちます。

   +++

### Campaign の設定とペイロードの問題 {#payload-issues}

[!BADGE &#x200B; 単一のユースケースとブロードキャストのユースケースの両方に適用 &#x200B;]{type=Positive}

プロファイルは有効なトークンを持って存在するが、ライブアクティビティが表示されない。 この原因として、次のことが考えられます。

* サーフェスまたはチャネルの設定が正しくありません。
* API ペイロード構造が正しくありません。
* `content-state` と `attributes` がiOS `ActivityAttributes` の実装と一致しません。
* 古い `timestamp` （更新/終了に不可欠）。

**ブロードキャストのユースケースに関するメモ**：キャンペーンは、（トランザクションではなく **API トリガーマーケティング** にする必要があります。 ペイロードでは、個々の `audience` の代わりに `profile` を使用します。 ブロードキャスト固有のペイロード構造については [&#x200B; この節 &#x200B;](#broadcast-config) を、完全な API 仕様については [Adobe Developer ドキュメント &#x200B;](https://developer.adobe.com/journey-optimizer-apis/references/messaging#operation/postIMAudienceMessageExecution) を参照してください。

#### 事前確認

* キャンペーンは **API トリガートランザクション** （単一）または **API トリガートランザクション** （ブロードキャスト）、**ハイスループット** オプションは、ライブアクティビティと互換性がないので **有効にしない** でください。
* [&#x200B; 上記のシナリオ &#x200B;](#profile-issue) を使用して、プロファイルが存在し、トークンが正しく同期されていることを確認します。

#### デバッグ手順

1. &#x200B;
   +++ キャンペーンサーフェス設定の検証

   1. Journey Optimizerで、**Campaign** を開き、**アクション** メニューに移動します。
   1. **ライブアクティビティ設定** を確認します。 このサーフェスは、プロファイルの `appId` スト内の `liveActivityPushNotificationDetails` に一致するバンドル ID を使用して、iOS アプリ用に設定する必要があります。 例えば、プロファイルに `"appId": "com.example.myapp"` がある場合、サーフェスは同じアプリをターゲットにする必要があります。
   1. キャンペーン設定の **アクティビティタイプ** がプロファイルの `attributeType` の `liveActivityPushNotificationDetails` と完全に一致することを確認します。 例えば、プロファイルに `"attributeType": "FoodDeliveryLiveActivityAttributes"` がある場合、キャンペーンはこの同じアクティビティタイプを指定する必要があります。

      +++

1. &#x200B;
   +++API ペイロード構造の検証

   API を使用してキャンペーンを実行する場合は、ペイロードが正しい構造に従っていることを確認します。

   **単一ペイロード：**

   ```json
   {
     "campaignId": "your-campaign-id",
     "recipients": [{
       "type": "aep",
       "userId": "user@example.com",
       "namespace": "email",
       "context": {
         "requestPayload": {
           "aps": {
             "content-available": 1,
             "timestamp": 1756984054,
             "event": "start",
             "attributes-type": "FoodDeliveryLiveActivityAttributes",
             "content-state": { ... },
             "attributes": { ... }
           }
         }
       }
     }]
   }
   ```

   **一般的なペイロードの問題：**

   | フィールド | 要件 | 共通の問題 |
   |-|-|-|
   | `attributes-type` | キャンペーン活動タイプおよびプロファイル `attributeType` と一致する必要があります | 不一致または誤字 |
   | `campaignId` | アクティブ化されたキャンペーン ID と一致する必要があります | キャンペーン ID が間違っているか、見つかりません |
   | `content-available` | `1` である必要があります | 値がないか間違っています |
   | `event` | `"start"`、`"update"` または `"end"` でなければなりません | イベントタイプが無効です |
   | `timestamp` | 常に現在/最新の Unix エポック時間（秒単位）である必要があります | 古い/キャッシュされたタイムスタンプの使用 |
   | `userId` または `namespace` | AEPの既存のプロファイルと一致する必要があります | プロファイル識別子の不一致 |

   **重要：常に最新のタイムスタンプを使用する**

   * `timestamp` フィールドは **常に** 各 API 呼び出しが行われた時点での **現在の Unix エポック時間** （秒単位）である必要があります。
   * これは、**、**、`start` 特に `update`**すべてのイベントタイプ`end`** に適用されます。
   * **更新/終了リクエストへの影響**：古いまたは古いタイムスタンプを使用すると、更新リクエストと終了リクエストが失敗するか、デバイスで無視されます。
   * 以前のリクエストのタイムスタンプを再利用したり **キャッシュされた値を使用したりしないでくださ**。
   * API 呼び出しごとに新しいタイムスタンプを生成します。

   **オプションフィールド（すべてのイベントタイプ）:**

   * `requestId`: トラッキングの一意の ID （推奨）。
   * `alert`：通知用の `title` と `body` を持つオブジェクト（更新に注意を引くのに役立ちます）。

   **バージョン情報 `dismissal-date`:**

   * Unix エポック時間（秒）を格納するオプションのフィールド。
   * **必要な場合にのみ`event: "end"`** 用します。
   * ライブアクティビティをデバイスから自動的に削除するタイミングを指定します。
   * エンドユーザーでライブアクティビティが指定されない場合、ユーザーがアクティビティを閉じるまで、ライブアクティビティは表示されたままになります。
   * 将来のタイムスタンプ（`timestamp` より後）である必要があります。

     +++

1. &#x200B;
   +++ ペイロードとiOS実装の整合

   API ペイロードがiOS アプリの `ActivityAttributes` 実装と一致することを確認します。 Adobe SDK `LiveActivityAttributes` プロトコルはiOS `ActivityAttributes` を拡張するものであり、`liveActivityData` プロパティが必要です。

   **マッピングを検証：**

   1. `ActivityAttributes` は、Adobe `LiveActivityAttributes` プロトコルを実装する必要があります。 例：

      ```swift
      struct FoodDeliveryLiveActivityAttributes: LiveActivityAttributes {
       public struct ContentState: Codable, Hashable {
           var orderStatus: String
           var estimatedDeliveryTime: String
       }
      
       // Adobe SDK requirement
       var liveActivityData: LiveActivityData
      
       // Your custom attributes
       var restaurantName: String
      }
      ```

      **メモ** `liveActivityData` フィールドは、Adobe SDKで必須であり、すべての実装に含める必要があります。

   1. API ペイロードは、iOS構造を反映する必要があります。

      ```json
      {
        "aps": {
           "event": "start",
           "timestamp": 1756984054,
           "attributes-type": "FoodDeliveryLiveActivityAttributes",
           "content-state": {
           "orderStatus": "Preparing",
           "estimatedDeliveryTime": "20 mins"
        },
        "attributes": {
          "liveActivityData": {
            "liveActivityID": "order-12345"
          },
          "restaurantName": "Pizza Palace"
        }
        }
      }
      ```

   **検証チェックリスト：**

   * すべての `ContentState` フィールドを `content-state` に含めます（すべてのイベントタイプで必須）。
   * 次のようなすべての `LiveActivityAttributes` フィールドを `attributes` に含める（開始イベントのみ）:
      * `liveActivityData` （必須。通常、`liveActivityID` または類似の識別子を含む）
      * 構造体からのすべてのカスタムフィールド
   * フィールド名を完全に一致させます（大文字と小文字を区別）。
   * データタイプ（文字列、整数、ブール、ネストされたオブジェクト）を一致させます。
   * ネストされたオブジェクト構造を保持します。

   **よくある間違い：**

   | 問題 | 影響 | 修正 |
   |-------|--------|-----|
   | 属性に `liveActivityData` がありません | ライブアクティビティが開始されない | 常に開始イベント `liveActivityData` オブジェクトを含める |
   | 開始イベントに必須フィールドがありません | ライブアクティビティが開始されない | iOS構造体からすべてのフィールドを追加します。 |
   | フィールド名が間違っています（誤字/大文字と小文字を区別） | フィールドが無視されたか、解析エラーが発生しました | iOSのフィールド名と完全に一致 |
   | データタイプが正しくありません | 解析エラー | iOS データタイプの一致 |
   | ネストされたオブジェクトがありません | 不完全なデータ | ネストされたすべての構造を含める |
   | 更新/終了に `attributes` を含める | 不要ですが、通常、無視されます | 開始イベントに `attributes` のみを含める |
   | 更新/終了時の古いタイムスタンプ | デバイスで更新/終了が無視されました | 常に新しいタイムスタンプを生成 |

   その他の例については、[&#x200B; ライブアクティビティのページを作成 &#x200B;](create-mobile-live.md) を参照してください。

   +++

1. &#x200B;
   +++ Assuranceを使用したテスト

   Assuranceを使用して API の実行とペイロード配信を確認します。

   1. Assurance セッションを開きます。
   1. API 呼び出しを実行して、ライブアクティビティをトリガーします。
   1. **イベントリスト** で、次の項目を確認します。
      * キャンペーン実行イベント。
      * ライブアクティビティ配信イベント。
      * ペイロード検証エラーイベント。
   1. イベントペイロードを確認して次の項目を確認します。
      * ペイロードが正しく処理されました。
      * 検証エラーは発生しませんでした。
      * ライブアクティビティが APN に送信されました。

      +++

### 配信の失敗とエラー分析

[!BADGE &#x200B; 単一のユースケースとブロードキャストのユースケースの両方に適用 &#x200B;]{type=Positive}

このシナリオでは、以前のチェックがすべて合格しています。

* [&#x200B; 有効なライブアクティビティのプッシュトークン &#x200B;](#profile-issue) を持つプロファイルが存在する
* キャンペーンが正しく設定されている [&#x200B; 適切なペイロードで設定されている &#x200B;](#payload-issues)
* [&#x200B; 更新トークンは同期されます &#x200B;](#token-not-synced) （更新/終了イベントの場合、単一のユースケースのみ）

ただし、ライブアクティビティが、期待どおりに表示、更新、終了しません。 この問題は、Adobe配信システムレベルで発生する場合や、プッシュ通知サービスプロバイダー（APN）で発生する場合があります。

**ブロードキャストのユースケースに関する注意**：レポートには、すべてのオーディエンスメンバーにわたる指標が表示されます。 成功するプロファイルもあれば、失敗するプロファイルもあります。

**チェック前**

* **検証済みの以前のシナリオ：**
   * プロファイルは正しい `liveActivityPushNotificationDetails` で存在します
   * キャンペーンサーフェスとアクティビティのタイプが正しい
   * API ペイロードは現在のタイムスタンプで有効です
   * 更新トークンが同期されます（更新/終了イベント用）

* **API 呼び出しが確認されました：**

   * API 呼び出しで HTTP 200 が返されました（成功）
   * キャンペーン ID と受信者の詳細が正しい

#### デバッグ手順

1. &#x200B;
   +++ キャンペーンレポートの確認

   1. **ライブアクティビティキャンペーン** に移動します。
   1. 「**レポート**」ボタンをクリックします。
   1. **すべての時間レポートを表示** を選択します。
   1. 以下の節を確認してください。

      1. **送信統計** 指標を確認して、配信の成功を把握します。

         | 指標 | 意味 | 探すもの |
         |-|-|-|
         | ターゲット | オーディエンスに適合したプロファイルの数 | テストプロファイルを含める必要があります |
         | 送信数 | 試行されたプッシュ通知の合計 | API 呼び出しと一致する必要があります |
         | 配信済み | デバイスに正常に配信されました | 送信と比較して成功率を確認 |
         | エラーを送信 | 送信に失敗したプッシュ通知 | 数値が大きい |
         | 除外の送信 | Adobe Journey Optimizerによって除外されるプロファイル | プロファイルが除外されたかどうかを確認する |

      1. 送信エラー > 0 の場合は、**エラー理由** 表で特定のエラーコードとメッセージを確認します。

         | 一般的なエラー | 意味 | 解決策 |
         |-|-|-|
         | 無効なトークン | プッシュトークンが無効か、期限切れです | デバイスからのライブアクティビティトークンの再登録 |
         | トークンが見つかりません | プロファイルに関連付けられた有効なトークンがありません | `liveActivityPushNotificationDetails` が存在することを確認 |
         | APNs rejected | Apple Push Notification サービスがプッシュを拒否しました | APNs 証明書、バンドル ID、環境を確認 |
         | ネットワークタイムアウト | APN に到達できません | 一時的な問題。API 呼び出しを再試行してください |

      1. **除外を送信** > 0 の場合は、**除外された理由** テーブルを確認します。

         | 一般的な除外 | 意味 | 解決策 |
         |-|-|-|
         | プロファイルがオプトアウトされました | ユーザーは通知をオプトアウトしています | プロファイル同意ステータスの確認 |
         | トークンブロックリストに加える | 無効としてマークされたトークン | トークンの再登録またはブロックリストステータスの確認 |
         | プロファイルが適格ではありません | プロファイルがキャンペーンの条件を満たしていません | キャンペーンオーディエンスルールのレビュー |

   詳しくは、[&#x200B; ライブアクティビティキャンペーンレポートページ &#x200B;](../reports/campaign-global-report-cja-activity.md) を参照してください。

   +++

1. &#x200B;
   +++ プロファイルでメッセージフィードバックイベントを確認

   1. Journey Optimizerで **顧客**/**プロファイル** に移動します。
   1. プロファイルを検索して開きます。
   1. 「**イベント**」タブを選択します。
   1. `eventType = "message.feedback"` を使用してイベントをフィルタリングまたは検索します。
   1. ライブアクティビティの `liveActivityID` と `event` のタイプに一致するフィードバックイベントを探します。
   1. 次の主要フィールドを確認します。

      | フィールド | 可能な値 | 意味 |
      |-|-|-|
      | `feedbackStatus` | `sent`、`error`、`denylist` | サービスプロバイダーからの配信結果 |
      | `serviceProvider` | `apns/apnsSandbox` | iOS ライブアクティビティの APN にする必要がある |
      | `errorCode` | 数値コードまたは `null` | 失敗した場合の APN 固有のエラーコード |
      | `errorMessage` | エラーの説明または `null` | 人間が読み取れるエラーメッセージ |

   1. **`feedbackStatus: "error"` の場合：**
      * `errorCode` と `errorMessage` で特定の APN エラーを確認します
      * 一般的な APN エラーには、期限切れのトークン、無効な証明書、間違ったバンドル ID が含まれます

   1. **フィードバックイベントが見つからない場合：**
      * プッシュ通知が試行されていない可能性があります
      * 上記の手順 1 で、プロファイルがキャンペーンレポートで除外されたかどうかを確認します。

      +++

1. &#x200B;
   +++ Assuranceでの APN に対するライブアクティビティ配信の検証

   1. Assurance セッションを開きます。このセッションは、API 呼び出し中にアクティブになる必要があります。
   1. API 呼び出し（開始、更新、終了）を実行します。
   1. **イベントリスト** で、ライブアクティビティ配信イベントを探します。
   1. APN プッシュ配信に関連するイベントを検索します。
   1. 次の指標を確認します。
      * **APN へのプッシュリクエスト**:AdobeがApple サーバーにプッシュを送信したことを確認します
      * **APNs response**:APN がプッシュを承認したか拒否したかを表示します
      * **配信ステータス**：成功または失敗の指標
   1. 問題が見つかった場合は、次の一般的な APN 配信の問題を参照してください。

      | 問題 | Assuranceの症状 | 解決策 |
      |-|-|-|
      | APNs 証明書の有効期限が切れました | 認証エラー | 新しい APN 証明書の更新とアップロード |
      | 間違った環境（開発と実稼動） | トークン不一致エラー | 証明書がアプリのビルドの種類と一致することを確認します |
      | バンドル ID が一致しません | 無効なバンドル識別子 | 証明書バンドル ID がアプリに一致することを確認します |
      | 期限切れのトークン | APN からの InvalidToken エラー | ライブアクティビティトークンの再登録 |
      | レート制限 | リクエストが多すぎます | API 呼び出し頻度の削減 |

      +++

1. &#x200B;
   +++ 追加の診断チェックに進む

   1. Campaign レポートのライブアクティビティのライフサイクル指標を確認します。

      キャンペーンレポートの「**ライブアクティビティのライフサイクル**」セクションを確認します。

      | 指標 | チェック対象 |
      |-|-|
      | リモート開始 | API トリガー開始数を表示する必要があります |
      | アップデート | 更新イベント数を表示 |
      | 終了 | 終了イベント数を表示 |
      | 合計数 | ライブアクティビティのイベントボリューム全体 |

      これらの指標がゼロであるか、API 呼び出しと一致しない場合は、Adobeと APN の間の配信に問題があります。

   1. Adobeに配信が成功したと表示されたが、デバイスにライブアクティビティが表示されない場合：

      * iOS デバイスのログでライブアクティビティエラーを確認します。
      * アプリがフォアグラウンドまたはバックグラウンド （終了していない）にあることを確認します。
      * デバイスがネットワークに接続されていることを確認します。
      * 複数のデバイスでテストを実行して、デバイス固有の問題を除外します。
      * iOSのバージョンが 16.1 以降であることを確認します。

      +++

1. &#x200B;
   +++ Adobe サポートへのエスカレーション

   すべての手順を完了しても問題が解決しない場合は、Adobe カスタマーケアに連絡して、次の内容をお知らせください。

   **必要な情報：**

   * キャンペーン ID と名前
   * プロファイル名前空間と ID
      * API ペイロードからの `liveActivityID`
   * API 呼び出しのタイムスタンプ
   * 以下のスクリーンショット
      * キャンペーンレポート （送信統計、エラー理由、除外された理由）
      * プロファイルイベント （`liveActivity.updateToken`、`message.feedback`）
      * 配信イベントを示すAssurance セッション
   * API リクエストペイロードの完了
   * APN 証明書の詳細（有効期限、環境、バンドル ID）

     +++

## 単一固有のシナリオ

### ライブアクティビティの更新トークンが同期されていません{#token-not-synced}

ライブアクティビティはデバイスで正常に開始されますが、後続の `update` または `end` API 呼び出し（HTTP 200 を返す）でライブアクティビティを更新または解除できません。 この問題は、**ライブアクティビティの更新トークン** がAdobeのシステムに正しく同期されていない場合に発生します。

**更新トークンについて**

デバイスでライブアクティビティが開始すると、iOSはその特定のライブアクティビティインスタンスに対して一意の更新トークンを生成します。 このトークンは、次の場合に必要です。

* ライブアクティビティに対する更新の送信
* ライブアクティビティのリモート終了

各ライブアクティビティインスタンスには、独自の一意の更新トークンがあります。 Adobeが更新イベントと終了イベントを配信するには、このトークンが必要です。

**予期される動作**

更新イベントと終了イベントが機能するには、次の手順を実行する必要があります。

1. ライブアクティビティは、デバイスで正常に開始します。
1. デバイスが、そのライブアクティビティインスタンスの更新トークンを生成します。
1. Mobile SDKは、更新トークンをキャプチャしてAdobeに送信します。
1. 更新トークンが同期され、Adobe システムに保存されます。
1. 更新/終了の後続の API 呼び出しは、配信にこのトークンを使用します。

**事前チェック：**

* **ユーザー権限**：デバイスで初めてライブアクティビティが開始すると、iOSに「[ アプリ名 ] によるライブアクティビティのアップデート提供を許可しますか？」というシステムプロンプトが表示されます。 更新トークンを生成して同期するには、ユーザー **「許可」をタップする必要があります**。 ユーザーが「許可しない」をタップすると、更新トークンは作成されず、更新/終了リクエストが失敗します。 これは、アプリごとに 1 回限りの権限です。
* **プロファイルとキャンペーンの検証**:[&#x200B; シナリオ 1](#profile-issue) と [&#x200B; シナリオ 2](#payload-issues) を完了し、プロファイル、トークン、キャンペーン設定が正しいことを確認します。

#### デバッグ手順

1. &#x200B;
   +++ Assuranceでの更新トークン同期の検証

   1. Assurance セッションを開きます。
   1. デバイスでライブアクティビティが開始されたときに、セッションがアクティブだったことを確認します。
   1. `eventType = "liveActivity.updateToken"` を使用してイベントをフィルタリングまたは検索します。
   1. イベントを選択し、ペイロードを調べます。

      * `token` フィールドに有効な更新トークン文字列が含まれていることを確認します。
      * `liveActivityID` がライブアクティビティインスタンスと一致することを確認します。
      * `activityType` が `attributes-type` と一致することを確認します。

   1. イベントが見つからない場合：

      * 更新トークンが生成されていないか、SDKによってキャプチャされていません。
      * ユーザーにライブアクティビティ権限が付与されているかどうかを確認します。
      * デバイスでライブアクティビティが実際に正常に開始されたことを確認します。
      * 更新トークンをキャプチャするために、モバイル SDKが適切に統合されていることを確認します。

   1. イベントが見つかった場合は、手順 2 に進みます。

      +++

2. &#x200B;
   +++ プロファイルイベントでの更新トークンの検証

   1. Journey Optimizerで **顧客**/**プロファイル** に移動します。
   1. プロファイルを検索して開きます。
   1. 「**イベント**」タブを選択します。
   1. `liveActivity.updateToken` イベントを探します。
   1. イベントの詳細を確認します。

      * タイムスタンプが最近のものであることを確認します（ライブアクティビティを開始した時点と一致）。
      * `token` と `liveActivityID` が存在することを確認します。
      * `activityType` が正しいことを確認します。

   1. イベントがプロファイルに見つからない場合：

      * 更新トークンイベントが、まだプロファイルに取り込まれていない可能性があります。
      * 5～10 分待ってから、もう一度確認してください。
      * 15 分後にもまだ見つからない場合は、イベント取り込みに問題がある可能性があります。

   1. イベントが見つかった場合、更新トークンは同期されています。 手順 3 に進むことができます。

      +++

3. &#x200B;
   +++ Assuranceでのライブアクティビティ配信イベントの確認

   1. Assurance セッションで、API 呼び出しの更新または終了を実行します。
   1. **イベントリスト** で、ライブアクティビティ配信イベント （APN プッシュイベント）を探します。
   1. 以下を示すイベントを確認します。
      * プッシュ通知が APN に送信されました。
      * APN からの応答（成功またはエラー）。
      * 配信の確認。
   1. APN 配信イベントが存在する場合：プッシュ通知が送信されました。 それでもデバイスが更新されない場合、問題はデバイス側にある可能性があります（アプリがプッシュを処理しない、ネットワークの問題など）。
   1. APN 配信イベントがない場合：更新トークンが正しく保存されていないか、Adobe System のプロファイルに関連付けられていない可能性があります。
   1. エラーイベントが存在する場合：特定の失敗理由（無効なトークン、APN が却下されたなど）について、エラーの詳細を調べます。

      +++

## ブロードキャスト固有のシナリオ

### ブロードキャストキャンペーンの設定とペイロードの問題{#broadcast-config}

ここでは、ブロードキャストのライブアクティビティに特有のシナリオで、単一キャンペーンとは異なるデバッグ方法が必要な場合のトラブルシューティングについて説明します。

プロファイルに有効なトークンがあるが、ライブアクティビティがオーディエンスメンバーに対して表示、更新または期待どおりに動作しない場合、この問題は通常、次のいずれかに起因します。

* キャンペーンが API トリガーマーケティングとして設定されていない。
* API ペイロードで使用しているブロードキャスト構造が正しくありません（`audience` または `input-push-channel` がありません）。
* `content-state` フィールドと `attributes` フィールドがiOS `ActivityAttributes` の実装と一致しません。
* Apple開発者ポータルで `input-push-channel` が正しく作成されませんでした。

このトラブルシューティングシナリオは、ブロードキャストキャンペーンのすべてのライブアクティビティイベント（`start`、`update`、`end`）に適用されます。

**事前チェック：**

* **キャンペーンタイプ**:
   * キャンペーンが API トリガーマーケティングとして作成されていることを確認します（ブロードキャスト/オーディエンスベースのキャンペーンに必要）。
   * オーディエンスがキャンペーン設定で定義されていることを確認します。
* **プロファイルとトークンの検証**：オーディエンスから複数のプロファイルをサンプル化し、有効な `liveActivityPushNotificationDetails` を持っていることを検証します。 検証手順について詳しくは、[&#x200B; シナリオ 1](#profile-issue) を参照してください。

#### デバッグ手順

1. &#x200B;
   +++ キャンペーンオーディエンス設定の検証

   1. **API トリガーマーケティングキャンペーン** をJourney Optimizerで開きます。
   1. **オーディエンス** セクションに移動し、次のことを確認します。
      * キャンペーンのオーディエンスが選択された。
      * オーディエンス ID は、API ペイロードで使用されたものと一致します。
      * オーディエンスには、想定されるプロファイルが含まれています。
   1. 「**アクション**」セクションに移動します。
   1. **ライブアクティビティ設定** を確認します。
      * 設定は、適切なバンドル ID を持つiOS アプリケーションに対して設定する必要があります。
      * アクティビティタイプは、API ペイロードの `attributes-type` と一致する必要があります。 例えば、ペイロードに `"attributes-type": "AirplaneTrackingAttributes"` が含まれる場合、キャンペーンはこの同じアクティビティタイプを指定する必要があります。

      +++

1. &#x200B;
   +++ ブロードキャスト API ペイロード構造を検証

   ブロードキャストペイロード構造は単一キャンペーンとは異なります。 ペイロードが正しいブロードキャスト形式に従っていることを確認します。

   **ブロードキャストの必須フィールド：**

   ```json
   {
     "campaignId": "878a11d4-b519-47bd-8313-fecfee19857b",
     "audience": {
       "id": "8c3dbdea-2957-401f-acf0-3966fba1601e"
     },
     "context": {
       "requestPayload": {
         "aps": {
           "input-push-channel": "FEt0NgvLEfEAAOqA6AXdIQ==",
           "content-available": 1,
           "timestamp": 1771829292,
           "event": "update",
           "attributes-type": "AirplaneTrackingAttributes",
           "content-state": { ... },
           "attributes": { ... }
         }
       }
     }
   }
   ```

   **一般的なペイロードの問題：**

   | フィールド | 要件 | 共通の問題 |
   |-|-|-|
   | `campaignId` | アクティブ化されたマーケティングキャンペーン ID と一致する必要があります | キャンペーン ID またはトランザクションキャンペーンの使用が正しくありません |
   | `audience.id` | AEPの既存のオーディエンスと一致する必要があります | 間違ったオーディエンス ID またはオーディエンスが存在しません |
   | `input-push-channel` | ブロードキャストに必須 – このブロードキャストインスタンスの一意の ID | `channelID` の `liveActivityData` が見つからないか一致しません |
   | `timestamp` | 常に現在/最新の Unix エポック時間（秒単位）である必要があります | 古い/キャッシュされたタイムスタンプの使用 |
   | `event` | `"start"`、`"update"` または `"end"` でなければなりません | イベントタイプが無効です |
   | `attributes-type` | キャンペーン活動タイプと一致する必要があります | 不一致または誤字 |
   | `content-available` | `1` である必要があります | 値がないか間違っています |

   **重要なブロードキャスト固有のフィールド：**

   * **`input-push-channel`**：
      * すべてのブロードキャスト ライブ アクティビティに必須です。
      * この特定のブロードキャストインスタンスの一意の ID として機能します。
      * オーディエンス内のすべてのプロファイルは、このチャネルにリンクされたライブアクティビティを受け取ります。
      * は、`channelID` の `liveActivityData.channelID` と一致する必要があります（手順 3 を参照）。
      * クライアントが、Apple開発者ポータルで `appID` 用に作成する必要があります。
      * そのアプリでライブアクティビティのブロードキャストに使用できるのは、特定のア `appID` ット用に作成されたチャネルのみです。

   * **`audience.id`**：
      * Adobe Experience Platformで作成された有効なオーディエンスセグメントを参照する必要があります。
      * このオーディエンス内のすべてのプロファイルが、ライブ アクティビティのターゲットになります。
      * オーディエンスをアクティベートし、有効な `liveActivityPushNotificationDetails` を持つプロファイルを含める必要があります。

   **常に最新のタイムスタンプを使用する：**

   * `timestamp` フィールドは、すべての API 呼び出しの現在の Unix エポック時間（秒単位）である必要があります。
   * この要件は、すべてのイベントタイプ（`start`、`update`、`end`）に適用されます。
   * **更新/終了に関して重要**：古いタイムスタンプを使用すると、更新と終了のリクエストが失敗する。
   * ブロードキャスト API 呼び出しごとに新しいタイムスタンプを生成します。

   **オプションフィールド：**

   * `dismissal-date`：自動解除の Unix エポック時間（`end` イベントにのみ関連）
   * `alert`：通知用の `title` と `body` を持つオブジェクト

   API の仕様について詳しくは、[Adobe Journey Optimizer Messaging API ドキュメント &#x200B;](https://developer.adobe.com/journey-optimizer-apis/references/messaging) を参照してください。

   +++

1. &#x200B;
   +++ content-state、attributes、input-push-channel をiOSの実装に合わせます

   ペイロードフィールドがiOS アプリの `ActivityAttributes` 実装と一致すること、および `input-push-channel` が `channelID` の `liveActivityData` と一致することを確認します。

   1. iOS ActivityAttributes の定義を確認します。

   カスタム `ActivityAttributes` 構造体は、次のAdobe `LiveActivityAttributes` プロトコルを実装する必要があります。

   ```swift
   struct AirplaneTrackingAttributes: LiveActivityAttributes {
    public struct ContentState: Codable, Hashable {
        var journeyProgress: Int
    }
   
    // Adobe SDK requirement
    var liveActivityData: LiveActivityData
   
    // Your custom attributes
    var arrivalAirport: String
    var departureAirport: String
    var arrivalTerminal: String
   }
   ```

   1. iOS フィールドをブロードキャスト API ペイロードにマッピングします。

   すべてのイベントに、`attributes` と `content-state` の両方を含めます。

   ```json
         {
         "aps": {
          "input-push-channel": "FEt0NgvLEfEAAOqA6AXdIQ==",
          "event": "start",
          "timestamp": 1771829292,
          "attributes-type": "AirplaneTrackingAttributes",
          "content-state": {
            "journeyProgress": 0
          },
          "attributes": {
            "arrivalAirport": "DEL",
            "departureAirport": "MUM",
            "arrivalTerminal": "T1",
            "liveActivityData": {
              "channelID": "FEt0NgvLEfEAAOqA6AXdIQ=="
            }
          }
         }
         }
   ```

   **重大：`input-push-channel` は`channelID`** に一致する必要があります

   * `input-push-channel` のルートにある `aps` 値は、`channelID` の `liveActivityData` と完全に一致する必要があります。
   * 上記の例では、両方の値が `"FEt0NgvLEfEAAOqA6AXdIQ=="` です。
   * この一致により、ブロードキャストインスタンスがライブアクティビティデータにリンクされます。
   * 不一致により配信エラーが発生する。

   **主な検証ポイント：**

   * すべてのイベントタイプに対して、`ContentState` のすべての `content-state` フィールドを含めます。
   * 開始イベントにのみ、すべてのカスタム `LiveActivityAttributes` フィールドを `attributes` に含めます。
   * 開始イベントの場合、`liveActivityData.channelID` は `input-push-channel` に一致する必要があります。
   * フィールド名は大文字と小文字が区別され、完全に一致する必要があります。
   * データタイプは一致する必要があります（文字列、整数、ブール値、ネストされたオブジェクトなど）。
   * 更新/終了イベントの場合は、元の開始イベントと同じ `input-push-channel` を使用します。

   **よくある間違い：**

   | 問題 | 影響 | 修正 |
   |-|-|-|
   | `input-push-channel` がありません | ブロードキャストが機能しない | 各ブロードキャストに一意のチャネル ID を追加 |
   | `input-push-channel` が `channelID` と一致しません | ライブアクティビティが開始されない | 両方の値が同じであることを確認します。 |
   | 更新/終了の `input-push-channel` が異なります | 更新/終了がライブアクティビティに到達しない | ライフサイクル全体で同じチャネル ID を使用 |
   | `liveActivityData.channelID` がありません | ライブアクティビティがブロードキャストにリンクされない | 開始イベントの属性に `channelID` を含める |
   | 開始イベントに必須フィールドがありません | ライブアクティビティが開始されない | iOS構造体からすべてのフィールドを追加します。 |
   | フィールド名が間違っています（誤字/大文字と小文字を区別） | フィールドが無視されたか、解析エラーが発生しました | iOSのフィールド名と完全に一致 |
   | 更新/終了時の古いタイムスタンプ | デバイスによって無視された更新/終了 | 常に新しいタイムスタンプを生成 |

   +++

1. &#x200B;
   +++ Assuranceを使用したテスト

   Assuranceを使用して API の実行とペイロード配信を確認します。

   1. オーディエンスの一部であるテストデバイスでAssurance セッションを開きます。
   1. ブロードキャスト API 呼び出しを実行します。
   1. **イベントリスト** で、以下を検索します。
      * キャンペーン実行イベント。
      * ライブアクティビティ配信イベント。
      * ペイロードの検証エラーを示すエラーイベント。
   1. イベントペイロードを調べて確認する：
      * ペイロードが正しく処理されました。
      * `input-push-channel` が存在する。
      * 検証エラーは発生しませんでした。
      * ライブアクティビティは、オーディエンスメンバー向けに APN に送信されました。

### プロファイルがオーディエンスまたは古いオーディエンススナップショットにありません

このシナリオでは、キャンペーンとペイロードは正しく設定されていますが、特定のプロファイルがライブアクティビティを受け取っていません。 これは通常、次の場合に発生します。

* プロファイルは、キャンペーンにリンクされたオーディエンスのメンバーではありません。
* オーディエンスはバッチオーディエンスで、プロファイルデータの古いスナップショットが含まれています。
* プロファイルのライブアクティビティトークンは最近追加されましたが、まだオーディエンスのスナップショットに反映されていません。

このトラブルシューティングシナリオは、特にオーディエンスベースのターゲティングを使用したブロードキャストキャンペーンに適用されます。

**オーディエンス評価について**

Adobe Experience Platformでは、プロファイルの更新がオーディエンスに反映されるタイミングを決定する、様々なオーディエンス評価方法を使用します。

| メソッド | 評価頻度 | データの鮮度 | 最適な用途 |
|-|-|-|--|
| バッチ | 1 日に 1 回（予定） | 最大 24 時間古くなっている可能性があります | 大きなオーディエンス、時間依存なし |
| ストリーミング | リアルタイム（プロファイル変更時） | 更新されたプロファイルのほぼリアルタイム | 時間に依存し、プロファイルの更新が必要 |

**事前チェック：**

* **キャンペーンとペイロードの検証**:
   * [&#x200B; このシナリオ &#x200B;](#broadcast-config) のチェックを完了して、キャンペーンとペイロードが正しいことを確認します。
   * API ペイロードの `audience.id` がキャンペーン設定と一致することを確認します。
* **プロファイルが存在します**：プロファイルが有効な `liveActivityPushNotificationDetails` を持つAEPに存在することを確認します。

#### デバッグ手順

1. &#x200B;
   +++ プロファイルがオーディエンスに含まれていることを確認

   まず、ライブアクティビティを受け取る必要があるプロファイルが、実際にオーディエンスの一部であるかどうかを確認します。

   1. Adobe Experience Platformの **オーディエンス** に移動します。
   1. キャンペーンの `audience.id` を使用してオーディエンスを検索し、開きます。
   1. **参照** または **サンプルプロファイル** をクリックして、オーディエンスメンバーを表示します。
   1. 名前空間と ID 値を使用して、テストプロファイルを検索します。
   1. **プロファイルがオーディエンスに見つからない場合：**
      * プロファイルはオーディエンス条件またはセグメントルールを満たしていません。
      * オーディエンス定義を確認し、メンバーシップ要件を理解します。
      * プロファイルデータまたはオーディエンス定義を更新して、プロファイルを含めます。
      * オーディエンスの評価が完了するまで待ちます（手順 2 を参照）。
   1. **プロファイルがオーディエンスに見つかった場合：** 手順 2 に進んで、データの鮮度を確認します。

      +++

2. &#x200B;
   +++ オーディエンス評価のタイプとスケジュールを確認

   データの鮮度を決定する際に、オーディエンスでバッチ評価とストリーミング評価のどちらを使用するかを特定します。

   1. **オーディエンスの詳細** ページで、**評価方法** を確認します。
      * **バッチ**：スケジュールに従って 1 日に 1 回評価されます。
      * **ストリーミング**：プロファイルの更新が発生したときにリアルタイムで評価されます。
      * **Edge**: エッジの場所でリアルタイムに評価されます。

   評価方法に基づいて、適切なトラブルシューティング手順に従います。

   **オーディエンスがバッチ評価を使用する場合：**

   1. **バッチオーディエンスの制限について：**
      * バッチオーディエンスは 1 日に 1 回（通常は夜間）評価されます。
      * オーディエンスのスナップショットは、最大 24 時間前になる場合があります。
      * プロファイルが最近登録したライブアクティビティトークンの場合、それらのトークンは現在のスナップショットにない可能性があります。
      * プロファイルの更新は、次のバッチ評価まで反映されません。

   1. **最後の評価が実行された日時を確認する：**
      * オーディエンスの詳細で、「最後の評価 **タイムスタンプを探し** す。
      * このタイムスタンプの後にプロファイルの `liveActivityPushNotificationDetails` が更新された場合、オーディエンスには古いデータが含まれています。

   1. **古いデータの解決：**
      1. **オプション 1：スケジュールされたバッチ評価を待機**
         * 次のバッチ評価には、更新されたプロファイルデータが含まれます。
         * これは 1 日に 1 回自動的に行われます。
         * 緊急ではないシナリオに最適です。

      1. **オプション 2:トリガーのオンデマンドオーディエンス評価**
         1. AEPの **オーディエンス** に移動します。
         1. オーディエンスを選択します。
         1. **今すぐ評価** または **オンデマンドでアクティベート** をクリックします。
         1. 評価が完了するまで待ちます（オーディエンスサイズによっては、数分から数時間かかる場合があります）。
         1. プロファイルがオーディエンススナップショットのデータを更新したことを確認します。
         1. ブロードキャスト API 呼び出しを再試行します。

   **オーディエンスがストリーミング評価を使用する場合：**

   1. **ストリーミングオーディエンスの動作について：**
      * ストリーミングオーディエンスは、プロファイルの更新が発生したときにリアルタイムで評価されます。
      * **新規プロファイル**：セグメント条件を満たす場合は、作成後すぐに選定されます。
      * **更新されたプロファイル**：更新された直後に選定または選定解除します。
      * **既存の未変更のプロファイル**：更新が行われない限り、再評価されません。

   1. **問題の特定：**
      * プロファイルが既に存在してセグメント条件を満たしているが、そのプロファイルを更新しない場合、新しく作成したストリーミングオーディエンスに追加できない可能性があります。
      * トリガーの再評価を行うには、プロファイルが更新（属性の変更）を受け取る必要があります。

   1. **問題を解決する：**
      * **新規プロファイルの場合**：条件が満たされた場合に自動的に選定されます。 アクションは必要ありません。
      * **最近の更新を含まない既存プロファイルの場合：**
         * プロファイルに小規模な更新（タイムスタンプフィールドの更新など）を行います。
         * このトリガーは、評価をストリーミングし、プロファイルをオーディエンスに追加します。
         * 代替：既存のプロファイルにバッチオーディエンスまたはエッジオーディエンスを使用します。

      +++
