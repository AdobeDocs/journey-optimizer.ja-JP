# コンテンツの実験について学習します。 {#get-started-experiment}

>[!AVAILABILITY]
>
>コンテンツ実験機能は、現在のところ、組織のセットに対してのみ使用することができます (利用可能な機能が限られています)。 詳しくは、アドビの担当者にお問い合わせください。

## コンテンツの実験について

コンテンツの実験を行うと、キャンペーンのアクションのコンテンツを最適化することができます。

実験とは、オンラインテストのコンテキストにおけるランダムな試行のセットで、ランダムに選択された複数のユーザーが、特定のメッセージのバリエーションに公開され、別のランダムに選択されたユーザー群が別の処理を行うことを意味します。 メッセージを送信した後で、関心のある結果メトリックを測定できます。例えば、電子メールまたはクリックのように表示されます。

## 実験を実行する理由

![](assets/content_experiment_schema.png)

実験により、メトリックスの改善につながる変更を切り分けることができます。 上記の図に示すように、ランダムに選択したユーザーは、各処理グループに対して提示されます。これは、グループが同じ特性を共有することを意味します。 これにより、結果の違いは、受け取る処理の違いによっても解釈される場合があります。これにより、変更を加えた結果と関心のある結果との間に因果リンクを確立できます。

これにより、データを基準として、ビジネス上の目的を最適化することができます。

Adobe 旅のオプティマイザーによるコンテンツの実験については、次のようなアイデアをテストすることができます。

* **Subject 行** : 変更がトーンにどのように影響するか、または題名行のパーソナライズ度にどのような影響を与える可能性がありますか。
* **メッセージの内容** : 電子メールの視覚的なレイアウトを変更することができます。

## コンテンツの実験はどのように動作しますか。 {#content-experiment-work}

### ランダムな割り当て

アドビシステムズ社では、コンテンツの実験において、ビジター id の擬似ランダムハッシュを使用して、ユーザーが定義したいずれかの操作を実行対象とするユーザーに対して、ランダムな割り当てを行います。 ハッシングメカニズムにより、ビジターがキャンペーンに何度も入力した場合でも、その処理は決定的に実行されます。

詳しくは、MumurHash3 32-bit アルゴリズムを使用して、ユーザー id ストリングを1万バケットのいずれかにハッシュします。 各扱いに割り当てられたトラフィックを50% 使用しているコンテンツ実験において、バケット1–5000のユーザーに最初の処理が与えられます。また、バケット5001から1万のユーザーは、2番目の処理を受け取ることになります。 擬似ランダムハッシュが使用されているので、ユーザーが観察したビジターは、完全に50-50 ではない可能性があります。それにもかかわらず、スプリットはターゲットの分割率と同じように統計的になります。

コンテンツ実験を使用してすべてのキャンペーンを設定する際には、ランダム化アルゴリズム用にユーザー Id を選択する id 名前空間を選択する必要があることに注意してください。 これは実行アドレス ](../configuration/primary-email-addresses.md) とは関係 [ ありません。

### データ収集と分析

割り当て時に、つまり、メッセージが送信チャネルで送信されたとき、またはユーザーが受信チャネルでキャンペーンを入力したときに、適切なシステムデータセットに「割り当てレコード」が記録されます。 これによって、ユーザーに割り当てられた処理が記録されます。また、試行錯誤とキャンペーン id も記録されます。

客観的なメトリックは、次の2つの主なクラスにグループ化できます。

* ユーザーが電子メールを開く、リンクをクリックするなどの操作に直接反応する直接のメトリック。
* ユーザーが処理を実行した後に発生する、間接的または「フィルター」メトリックの測定値。

Adobe 旅のオプティマイザーがメッセージを追跡する直接的な目的のためのメトリックスについては、エンドユーザーの応答イベントに、自動的にキャンペーンおよび処理の識別子を使用してタグを設定します。これにより、応答メトリックと処理を直接関連付けることができます。 [トラッキング ](../email/message-tracking.md) について学習します。

![](assets/technote_2.png)

購買のような間接的な「下位にあるフィルター」の目的については、エンドユーザーのレスポンスイベントには、キャンペーンや取り扱いの識別子がありません。購入イベントは、メッセージが公開された後に発生するので、事前の処置を行うと、その購入は直接関連付けられることはありません。 このようなメトリックについては、次のような場合に、この処理がじょうご変換イベントの下に関連付けられます。

* ユーザー id は、&quot;割り当て&quot; および &quot;変換&quot; イベント時間と同じです。
* この処理を実行すると、処理の割り当ての7日以内に実行されます。

![](assets/technote_3.png)

Adobe の旅オプティマイザーは、高度な &quot;いつでも有効な&quot; 統計情報を使用してこの未処理のレポートデータを解釈し、実験レポートを解釈します。 詳しくは、このページ ](../campaigns/experiment-calculations.md) を [ 参照してください。

## 実験を実行するためのヒント

実験を実行する際には、次のベストプラクティスに従うことが重要です。 このような実験を実行するためのヒントを次に示します。

+ + + テストしようとしている変数を分離します。

テストの対象となる仮説を策定し、そのような仮説を可能な限り少ない変更に制限することによって、そのような結果が得られるかどうかを判断します。

例えば、「電子メールの件名のパーソナライズ」が高いオープンレートを使用しているかどうかを仮説にすることをお勧めします。 ただし、メッセージコンテンツやイメージに変更を加えた場合は、紛らわしい結果が得られることがあります。
+++

+ + + 適切なメトリックが使用されていることを確認します。

ターゲットとするメトリックを決定し、変更を行うことができるかどうかを判断してください。

例えば、メッセージ本文の内容を変更しても、電子メールを開くレートに影響することはまずありません。
+++

+ + + テストを適切な対象ユーザーサイズで実行します。または、

長期間テストを実行すると、操作の間に目標メトリック内の小さな差分が検出されるようにすることができます。 ただし、基準値が小さすぎると、より大きなサンプルサイズが必要になります。実験に含める必要があるユーザー数は、検出する結果サイズ、これによって目標指標の変化、拡大、偽陽性および偽否定的エラーに対する許容度によって異なります。 古典的な実験においては、サンプルサイズの [ 電卓 ](https://experienceleague.adobe.com/tools/calculator/testcalculator.html) {_blank} を使用して、テストの実行時間を決定することができます。
+++

+ + + 統計的不確実性について説明しています。

1000のユーザーが1回の操作を行って、変換レートが5% に設定されている場合は、 これは、すべてのユーザーが含まれている場合の実際の移行率ですか? 実際の変換レートはどのようになりますか。統計メソッドを使用すると、不確実性が不明確になります。 オンライン実験を実行する際に理解しておくべき最も重要な概念の1つは、実際の変換レートが、基本的な実際の変換レートと一致しているということです。 このような不確実性を定量化するには、自信を利用してください。
+++

+ + + を新しい hypotheses に追加し、継続的にテストします。

真の業務を実現するには、1回の実験だけにしてください。 このような場合は、新しい hypotheses を作成し、異なるセグメント上に異なる変更が加えられた新しいテストを作成して、それぞれの測定結果に影響を与えるようにします。
+++

## 実験結果の解釈 {#interpret-results}

![](assets/experimentation_report_3.png)

この節では、実験レポートについて説明し、表示される様々な統計量について理解する方法について説明します。

ここでは、コンテンツ実験の結果を解釈するためのガイドラインを示します。

結果の詳細な説明には、使用可能な証拠 (サンプルサイズ、変換率、信頼度など) がすべて含まれていることを考慮する必要があります。これは、決定的な宣言だけではありません。 結果がまだ決定的でない場合でも、ある操作が別の操作と異なる場合は、強力な証拠が得られます。

統計計算について理解するには、この [ ページ ](../campaigns/experiment-calculations.md) を参照してください。

### 1. 正規化されたメトリックスを比較します。 {#normalized-metrics}

2つの処理のパフォーマンスを比較する場合は、各処理に公開されたプロファイル数の違いを考慮して、正規化したメトリックスを常に比較する必要があります。

例えば、実験目的がに **[!UICONTROL Unique Opens]** 設定されている場合は、「1万」と表示されているプロファイル200で処理が記録されると、これは **[!UICONTROL Conversion Rate]** 2% の値を表します。 メトリックを開くなど、一意ではないメトリックについては、正規化されたメトリックはと表示 **[!UICONTROL Count per Profile]** されますが、値段が高い場合は、それによって **[!UICONTROL Total per Profile]** 標準化されたメトリックがに表示されます。

### 2. 信頼区間に焦点を絞る {#confidence-intervals}

プロファイルのサンプルについて実験を実行すると、指定された処理における換算率が、true を指定すると、基準となる変換レートの「推定」になります。

例えば、治療 A の数が **[!UICONTROL Conversion Rate]** 3% であるのに対し、その処理 b には2% の観測 **[!UICONTROL Conversion Rate]** があるとします。 このことを確認するには、まず、このような変換レートの不確実性を定量化する必要があります。

精度インターバルは、変換レートの概算値の不確実性の値を数値化するのに役立ちますが、広い信頼区間を使用すると不確実性が高くなります。 実験に追加されたプロファイルが増えると、間隔がより正確な見積もりを表すように小さくなります。 信頼区間は、観測データと互換性のある変換レートの範囲を表します。

これは、2つの処理の信頼度の間隔が非常に重複している場合、2つの処理に異なる変換レートが設定されていることを意味します。 ただし、2つの操作の信頼区間間に多くのオーバーラップがある場合は、2つの処理の間で同じ変換レートが発生する可能性が高くなります。

アドビでは、常に有効な信頼度インターバル、または信頼法を使用して95% を使用します。その結果は、実験中にいつでも安全に表示することができます。

### 3. 揚力について理解する {#understand-lift}

実験レポートの概要には、指定された処理の、ベースライン上の処理率の改善率の尺度が表示 **[!UICONTROL Lift over Baseline]** されます。 厳密には、指定された処理とベースラインとの間でのパフォーマンスの違いがベースラインのパフォーマンスを基準として、比率で表されます。

### 3. 信頼について理解する {#understand-confidence}

ここでは、各処理のパフォーマンスについて主に説明し **[!UICONTROL Confidence interval]** ますが、Adobe では、指定された処理がベースラインでの処理と同じであるという証拠については、probabilistic としても、自信が示されています。 精度が高いということは、ベースラインと非ベースの処理にパフォーマンスが同じであることを想定した証拠が少なくなることを意味します。 より正確には、表示されるのは、指定された処理とベースラインとの間で変換レートの差が小さくなる確率 (パーセンテージで表します) です。実際には、これまでの変換率が true になっていない場合があります。 P 値については、表示される精度は 1 ~ p 値です。

アドビでは、&quot;いつでも有効な&quot; 精度と、上で説明した信頼度と一致する有効な「p」値を使用しています。

### 4. 統計上の意味

実験が実行されると、計算結果は、与えられた処理とベースラインの基準となる本来の変換レートとパフォーマンスが同じであるという null 仮説が発生した場合に、統計的に有意とみなされます。

Adobe では、自信が95% を超えると、その実験が決定的に宣言されます。

## 実験実行後の作業

実験を実行した後で、次のようなフォローアップアクションがいくつかあります。

* **落札アイデアのデプロイ**

   最適な結果が得られるように、この受賞されたアイデアを展開するには、最良のパフォーマンスが得られる最高の処理をすべてのユーザーに提供するか、最高の実行処理を複製するような新規のキャンペーンを作成することができます。
   </br>動的環境では、一度にうまく機能することはありません。

* **フォローアップテストの実行**

   場合によっては、処置の違いを検出するための十分なプロファイルが含まれていないか、または、指定した処置が十分に異なることがあるために、実験結果が不確定になることがあります。

   テストした仮説が現在も関係がある場合は、それよりも大規模な対象ユーザーに対してフォローアップテストを実行するか、処理を修正して、わかりやすい結果が得られるようにすることをお勧めします。

* **詳細について詳しく解析**

   参加者にとって適切な処置が、他のユーザーにとっては最適でない場合があります。 様々なセグメントに対する処理がどのように行われているかについて詳しく解析することは、新しいテストのアイディアを生み出します。

   同様に、異なる評価基準により各処理のパフォーマンスを確認することにより、実験の内容を包括的に把握することもできます。

   >[!CAUTION]
   >
   >さらに多くの解析を行うと、擬似エフェクトを検出できる可能性が高くなり、偽陽性になります。
